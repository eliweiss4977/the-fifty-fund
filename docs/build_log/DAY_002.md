## The GPT Audit

Did something unusual: uploaded the entire repo as a zip to GPT-4 and asked it to tear the codebase apart.

GPT's verdict: *"Impressive for a Claude Code build. Would not trust it unattended with meaningful capital."*

Key criticisms:
- `docs/data.json` was acting as both public dashboard AND private source of truth
- No idempotency — a Railway restart could cause duplicate trades or posts
- Two validation systems running in parallel (old `validate_decision()` + new `risk_engine.py`)
- Risk guardrails lived in the AI prompt, not in deterministic code
- No append-only audit trail
- The agent trusted Claude's output too directly — model should be treated as untrusted suggestion engine

Best line from the review: *"Your moat is not better signals. Your moat is trust architecture for AI actions in public."*

## Architectural Migration

Brought GPT's feedback back to Claude. Instead of arguing, Claude accepted the core criticisms and shipped three new modules:

### Round 1 — New Modules
- **`agent/ledger.py`** — append-only JSONL event ledger with UTC timestamps
- **`agent/risk_engine.py`** — deterministic trade validation (max position %, cash buffer, PDT check, duplicate order window, min order value)
- **`agent/reconciliation.py`** — fetches actual Alpaca state every cycle, logs drift

### Round 2 — GPT Follow-Up
GPT reviewed the new modules and said: *"The architecture is improved. The implementation is only halfway through the migration."*

The problem: new modules existed but the old execution path still survived in parallel. Split-brain.

### Round 3 — Full Migration (in progress)
Final prompt issued to Claude Code to:
- Thread a single `cycle_id` through every action in a cycle
- Delete `validate_decision()` entirely — one risk path only
- Make reconciliation the sole source of portfolio truth
- Move internal state (`last_cycle_utc`, `last_outlook_date`) from `docs/data.json` to `data/state.json`
- Log every stage: CYCLE_START → RECONCILIATION → DECISION_PROPOSED → DECISION_VALIDATED → ORDER_SUBMITTED → ORDER_FILLED/REJECTED → POST_TELEGRAM → POST_X → DASHBOARD_UPDATED → CYCLE_END

Acceptance criteria: `git grep` must prove old paths are gone.

## Key Learnings

1. **Partial rigor is more dangerous than obvious roughness.** A system that looks solid but has split-brain architecture inspires false confidence.
2. **The model should never author the final order.** Claude proposes → risk engine validates → broker executes. Three separate authorities.
3. **Public state and private state must be separated.** Dashboard JSON is a view, not a source of truth.
4. **Cross-AI auditing works.** Using a rival AI to review your codebase surfaces blind spots the original builder missed.
5. **yfinance doesn't work on Railway IPs** — Alpaca's own historical data API is the reliable alternative.
